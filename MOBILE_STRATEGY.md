# Mobile Strategy: iOS & Android from Scala

## Goal

Ship the Twitch Stream Notifier on iOS and Android while keeping the codebase in Scala with pure FP. Minimize native Swift/Kotlin code.

---

## Approaches Evaluated

### 1. PWA (Progressive Web App) — Start Here

Make the existing Scala.js web app installable on mobile via a service worker and web app manifest.

**Code reuse:** 100%. No new code beyond a manifest file and service worker.

**What you get:**
- Installable home screen icon on both iOS and Android
- Full-screen app experience (no browser chrome)
- Push notifications on Android (Web Push API)
- Push notifications on iOS 16.4+ (Safari only, user must opt in)

**Limitations:**
- No background refresh — the app must be in the foreground to receive SSE events
- iOS push notification support is incomplete (no Chrome/Firefox on iOS, Safari-only)
- No App Store distribution without a native wrapper
- No access to Bluetooth, NFC, or contacts

**Effort:** 1-3 days. This is the validation step — test whether users actually want a mobile experience before investing in a native shell.

**Implementation:**
- Add `manifest.json` to `modules/frontend/` with app name, icons, theme color, `"display": "standalone"`
- Add a service worker for offline caching (static assets + API responses)
- Link the manifest from `index.html`
- The existing SSE notification system works as-is in the PWA

---

### 2. Scala.js + Capacitor — The Production Path

Compile Scala.js to JS (as today), then wrap it in [Capacitor](https://capacitorjs.com/) — a native shell that bridges JS to iOS/Android platform APIs.

**Code reuse:** ~95%. Your entire codebase (core models, circe codecs, Calico UI, http4s-dom client) compiles to JS unchanged. Capacitor doesn't care that the JS came from Scala.js.

**What non-Scala code is required:**
- Capacitor project scaffold (`capacitor.config.ts`, `package.json`)
- Xcode project files (generated by Capacitor, rarely touched manually)
- Android Studio project files (generated by Capacitor)
- Zero Swift/Kotlin for standard use cases

**What you write in Scala:**
- Thin `@js.native` facade traits over Capacitor's TypeScript plugin APIs (push notifications, background tasks, etc.). This is mechanical — roughly 20-50 lines per plugin.

**Native API access (via Capacitor plugins):**
- Push notifications (APNS on iOS, FCM on Android) — official plugin, production-grade
- Background tasks — official plugin, supports periodic background fetch
- Local notifications — official plugin
- App lifecycle events (foreground/background) — official plugin
- Camera, geolocation, haptics, share sheet — available if needed later

**Effort:** 1-2 weeks for a working app with push notifications.

**Implementation outline:**

```
twitch-app/
  modules/
    core/          (shared models — unchanged)
    frontend/      (Scala.js + Calico — unchanged)
    backend/       (http4s server — unchanged)
    mobile/        (NEW — Capacitor project)
      capacitor.config.ts
      ios/         (generated by `cap add ios`)
      android/     (generated by `cap add android`)
      src/
        scala/     (Scala.js facades for Capacitor plugins)
```

1. `sbt frontend/fullLinkJS` produces optimized JS
2. Copy JS + index.html + CSS into Capacitor's `www/` directory
3. `npx cap sync` copies web assets into native projects
4. `npx cap open ios` / `npx cap open android` to build in Xcode / Android Studio

**Key architectural change:** The mobile app would talk to the deployed backend (from the deployment plan) rather than localhost. The frontend already uses relative API paths, so this means configuring a base URL for API calls when running inside Capacitor.

**Push notifications shift:** Currently, notifications are delivered via SSE (requires the app/tab to be open). For mobile, the backend would additionally send push notifications via APNS/FCM when the StreamPoller detects new streams. This requires:
- A new `push_subscriptions` table in PostgreSQL (device token, user ID, platform)
- A `/api/push/register` endpoint for the mobile app to send its device token
- Server-side APNS/FCM integration (e.g., via an HTTP/2 client to Apple's push service)
- This builds on Step 7 (Web Push) from the deployment plan

### 3. Scala.js + React Native — If Native Feel Matters

Use [slinky-native](https://slinky.dev/) to write React Native components in Scala.js.

**Code reuse:** ~60%. Core models, circe codecs, and business logic transfer. The entire UI layer (Calico components) must be rewritten — React Native has no DOM, no CSS, and a different component model.

**Maturity:** slinky-native is maintained (v0.7.5, March 2025) but the Scala.js facade surface is thinner than the web side. You will encounter missing bindings for less common React Native libraries and need to write your own.

**When to choose this:** Only if WebView-based rendering (Capacitor) feels inadequate — e.g., complex gesture-heavy UIs, animations requiring 60fps native rendering, or platform-specific UI patterns (iOS navigation controllers, Android material components). For a notification-centric app like this, Capacitor is sufficient.

**Effort:** 4-8 weeks, dominated by rewriting the UI layer.

---

### Approaches Ruled Out

**Scala Native:** Does not support iOS or Android as compilation targets. The runtime (GC, threading) is not ported to mobile platforms. Not viable.

**Kotlin Multiplatform (KMP):** Scala JVM bytecode can be consumed on Android but not on iOS (KMP's iOS target uses Kotlin/Native, which doesn't read JVM bytecode). You'd end up rewriting shared logic in Kotlin, defeating the goal. KMP is a Kotlin-first solution, not a Scala bridge.

---

## Recommended Path

```
Phase 1: PWA (days)
  Add manifest + service worker to existing frontend.
  Validate that users want a mobile experience.
  Zero risk, zero native code.

Phase 2: Capacitor (1-2 weeks, after backend is deployed)
  Wrap the Scala.js frontend in Capacitor.
  Add push notification support (APNS/FCM).
  Ship to App Store / Play Store.
  ~95% Scala code reuse, minimal facade code.

Phase 3: Native enhancements (as needed)
  Background stream polling via Capacitor Background Task plugin.
  Deep links for sharing stream notifications.
  Offline support via service worker caching.
```

Phase 1 can happen now. Phase 2 depends on the backend being deployed (DEPLOY_PLAN.md Steps 1-6) since the mobile app needs a public API endpoint. Phase 3 is incremental.

---

## What Stays in Scala

| Component | Reused on Mobile? | Notes |
|-----------|-------------------|-------|
| Core models + circe codecs | Yes | Compiled to JS via Scala.js, used as-is |
| Calico UI components | Yes (Capacitor) / No (React Native) | Capacitor renders the same web UI |
| http4s-dom client calls | Yes | API calls work from WebView |
| StreamLogic (pure functions) | Yes | Tag filtering, notification transforms |
| Backend (http4s server) | N/A | Runs on server, mobile app calls it |

The only non-Scala code in the Capacitor path is project scaffolding and build configuration — no application logic in Swift or Kotlin.
